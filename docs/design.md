## 1. プロジェクト概要

SES企業のパートナー候補に対してお問い合わせを効率的に行うため、以下を自動化するシステム。

- Google検索によるSES企業一覧ポータルサイトの自動発見
- ポータルサイトからSES企業名・URLのスクレイピング
- 各企業サイトからお問い合わせフォームURLの自動取得
- Google Spreadsheetへの統合（重複排除済み）

管理基盤はGoogle Spreadsheetとし、GAS・Google Sheets APIは使用しない。すべての操作はPlaywright MCPによるブラウザ直接操作で完結する。

## 2. 全体アーキテクチャ

### データフロー図

```
[Google検索]
    |
    | ステップ1: ポータルサイト自動探索
    v
[ポータルサイト発見] ------> [ポータル管理シート（クロール済みURL管理）]
    |
    | ステップ2: 企業リスト収集
    v
[企業名・URL収集] ----------> [作業用シート（A列: 企業名, C列: URL）]
    |
    | ステップ3: お問い合わせURL取得
    v
[お問い合わせURL取得] -------> [作業用シート（D列: お問い合わせURL, E列: 取得状態）]
    |
    | ステップ4: シート統合
    v
[重複チェック（Spreadsheet関数）] --> [シート1（本番）: 新規企業のみ追加]
```

### コンポーネント構成

```
Claude Code (オーケストレーション・判断)
    |
    |-- Playwright MCP
    |       |-- Google検索操作
    |       |-- ポータルサイトスクレイピング
    |       |-- 企業サイトへのアクセス
    |       |-- Google Spreadsheet直接操作
    |
    |-- Python スクリプト（補助）
            |-- HTML解析
            |-- データ正規化
            |-- 重複チェックロジック補助
```

## 4. スプレッドシート構成

### 4.1 シート1（本番）

企業リストの本番データ。ステップ4の統合処理により更新される。

| 列 | 項目名 | 説明 |
|---|---|---|
| A | 企業名 | SES企業の正式名称 |
| B | フォーム送信 | チェックボックス。ヘッダーにCOUNTIF等で送信済み数を表示 |
| C | URL | 企業サイトのURL |
| D | お問い合わせURL | お問い合わせフォームのURL |
| E | 面談について | A/B等のランク（手動入力） |
| F | 概要 | 企業概要テキスト（手動入力） |
| G | メモ | 備考（手動入力） |
| H | 正規化企業名 | 重複チェック用ヘルパー列（Spreadsheet関数） |
| I | 正規化URL | 重複チェック用ヘルパー列（Spreadsheet関数） |

- B列ヘッダー: `=COUNTIF(B2:B,"TRUE")&"/"&COUNTA(B2:B)&" 送信済"` 等で自動集計
- 新規追加時: E列・F列・G列は空欄で初期化、B列はFALSE（未チェック）で初期化
- H列・I列は作業用シートのG列・I列と同じSpreadsheet関数で自動算出（重複チェック用）

### 4.2 作業用シート

スクレイピングで取得した一時データ。ステップ2・3で書き込まれ、ステップ4で参照される。

| 列 | 項目名 | 説明 |
|---|---|---|
| A | 企業名 | スクレイピングで取得した企業名 |
| B | フォーム送信 | FALSE（固定値で初期化） |
| C | URL | 企業サイトURL |
| D | お問い合わせURL | お問い合わせフォームURL |
| E | URL取得状態 | `URL済` / `取得失敗` / `アクセスエラー` |
| F | ソース | `new` またはポータルサイト名 |
| G | 正規化企業名 | 重複チェック用ヘルパー列（Spreadsheet関数） |
| H | 重複フラグ | TRUE=重複（統合除外）/ FALSE=新規（統合対象） |

### 4.3 カウントシート（既存）

進捗トラッキング用。既存の構成を維持する。

### 4.4 ポータル管理シート（新規追加）

クロール済みポータルを管理し、重複クロールを防ぐ。

| 列 | 項目名 | 説明 |
|---|---|---|
| A | ポータルURL | ポータルサイトのURL |
| B | ポータル名 | サイト名 |
| C | 最終クロール日 | 最後にクロールした日付（YYYY-MM-DD） |
| D | 取得企業数 | そのポータルから取得した企業数 |
| E | ステータス | `クロール済み` / `エラー` |

## 5. 全体フロー詳細

### ステップ1: ポータルサイト自動探索

**目的**: Google検索でSES企業一覧を掲載するポータルサイトを発見する。

**処理フロー**:

1. Playwright MCPでGoogle検索を実行（以下のクエリを順次または並行）
   - `SES 企業一覧`
   - `SES パートナー企業 一覧`
   - `SES会社 リスト`
   - `システムエンジニアリングサービス 企業一覧`
   - `SES企業 まとめ`
2. 検索結果のURLを取得し、企業一覧ページを持つポータルかをAIで判別
3. ポータル管理シートを参照し、既にクロール済みのURLはスキップ
4. 新規ポータルをポータル管理シートに記録（ステータス: クロール待ち）

**判別基準（AIによる評価）**:
- ページに複数企業のリストが存在するか
- 各企業に個別リンクが付与されているか
- SES・SE・IT企業に特化した一覧ページか

---

### ステップ2: 企業リスト収集

**目的**: 各ポータルサイトから企業名・企業URLを収集し、作業用シートに記録する。

**処理フロー**:

1. ポータル管理シートからクロール対象URLを取得
2. Playwright MCPで各ポータルにアクセス
3. ページのHTML構造をAIで解析し、企業名・URLを抽出
   - HTMLの固定パーサーに依存しない汎用解析
   - ポータルごとに異なるDOM構造に対応
4. ページネーションを検出し、全ページを走査
5. 取得データを作業用シートのA列（企業名）・C列（URL）に追記
6. ポータル管理シートのD列（取得企業数）・C列（最終クロール日）・E列（ステータス）を更新

**ページネーション対応**:
- 「次へ」「Next」「>」等のリンクを検出し、再帰的にページを取得
- ページ数上限なし（ポータル全体を網羅）

**アクセス間隔**: 各ページアクセス後に 1〜3秒 のウェイトを設定

---

### ステップ3: お問い合わせURL取得

**目的**: 作業用シートに記録された各企業サイトにアクセスし、お問い合わせフォームURLを特定する。

**処理フロー**:

1. 作業用シートのC列（企業URL）を順次読み取り
2. Playwright MCPで各企業サイトにアクセス
3. お問い合わせURLの探索（以下の優先順で判定）
   - ナビゲーション・フッターの「お問い合わせ」「contact」リンクを検出
   - URLパターンマッチング: `/contact`, `/inquiry`, `/form`, `お問い合わせ` 等
   - 実際にフォーム要素（`<form>`, `<input>`, `<textarea>`）が存在するページを優先
   - 複数候補がある場合はAIが最も適切なものを選択
4. 取得結果を作業用シートのD列（お問い合わせURL）・E列（URL取得状態）に記録

**取得状態の記録**:
- 取得成功: `URL済`
- フォーム未検出: `取得失敗`（D列は空欄）
- アクセスエラー: `アクセスエラー`（タイムアウト・404等）

**SPAサイト対応**: PlaywrightのヘッドレスブラウザによりJavaScript描画後のDOMを取得可能

**アクセス間隔**: 各企業サイトアクセス後に 1〜3秒 のウェイトを設定

---

### ステップ4: シート統合

**目的**: 作業用シートから重複を除いた新規企業のみをシート1（本番）に統合する。

**処理フロー**:

1. 作業用シートのH列（重複フラグ）を参照
2. FALSE（新規）の行のみをシート1に追加
3. シート1への追記内容
   - A列: 企業名
   - B列: FALSE（チェックボックス未チェック）
   - C列: 企業URL
   - D列: お問い合わせURL（取得失敗の場合は空欄）
   - E列・F列・G列: 空欄（手動入力用）
4. 統合完了後、作業用シートのデータはクリアせず保持する。次回実行時は既存データの下に追記し、重複フラグにより自動的に統合対象外となる。

**重複判定条件（OR条件で厳しめに判定）**:
- 企業名（表記ゆれ正規化後）がシート1に存在する
- URL（ドメイン正規化後）がシート1に存在する

## 6. 重複チェック設計

### 6.1 正規化ルール

**企業名の正規化**（取りこぼし防止のための表記ゆれ対応）:
- 「株式会社」「（株）」「(株)」を除去
- 全角・半角スペースを除去
- 大文字・小文字を統一（英字を含む場合）

**URLの正規化**:
- `https://` / `http://` を除去
- `www.` を除去
- 末尾スラッシュ `/` を除去

### 6.2 作業用シートに配置するSpreadsheet関数

**G列: 正規化企業名（ヘルパー列）**

```
=SUBSTITUTE(SUBSTITUTE(SUBSTITUTE(SUBSTITUTE(A2,"株式会社",""),"（株）",""),"(株)","")," ","")
```

**I列: 正規化URL（ヘルパー列）**

末尾スラッシュのみ除去し、中間パスを誤って結合しないよう REGEXREPLACE を使用する。

```
=REGEXREPLACE(REGEXREPLACE(REGEXREPLACE(C2,"^https?://",""),"^www\.",""),"/$","")
```

**H列: 重複フラグ**

部分一致による誤ヒットを防ぐため、EXACT関数による完全一致チェックを採用する。シート1側にもH列（正規化企業名）・I列（正規化URL）のヘルパー列を追加していることが前提。

```
=OR(
  SUMPRODUCT(EXACT(INDIRECT("シート1!H2:H"),G2))>0,
  SUMPRODUCT(EXACT(INDIRECT("シート1!I2:I"),I2))>0
)
```

- `TRUE`: 重複（シート1統合除外）
- `FALSE`: 新規（シート1統合対象）

### 6.3 シート1側の補助設計（任意）

- 条件付き書式: 重複検出行をハイライト表示
- B列ヘッダー: `=COUNTIF(B2:B,TRUE)&"件 送信済"` 等で送信進捗を自動集計

## 7. エラーハンドリング

| エラー種別 | 発生箇所 | 対応 | 記録先 |
|---|---|---|---|
| お問い合わせフォーム未検出 | ステップ3 | D列は空欄、E列に「取得失敗」を記録。シート1にはURL空欄で追加 | 作業用シートE列 |
| タイムアウト | ステップ3 | E列に「アクセスエラー」を記録してスキップ | 作業用シートE列 |
| 404/403等のHTTPエラー | ステップ3 | E列に「アクセスエラー」を記録してスキップ | 作業用シートE列 |
| ポータルスクレイピング失敗 | ステップ2 | ポータル管理シートのE列に「エラー」を記録してスキップ | ポータル管理シートE列 |
| JavaScript描画SPAサイト | ステップ2・3 | PlaywrightのWaitForLoadState等で描画完了を待機 | （特別な記録なし） |
| Google検索結果なし | ステップ1 | 次の検索クエリで再試行 | （特別な記録なし） |

## 8. 実行方法

### 一括実行コマンド（想定）

4ステップをセッション内で一括実行する。Claude CodeのSkillとして定義する想定。

```
/collect-ses-partners
```

または以下の形式でステップ個別実行も可能にする設計とする:

```
/collect-ses-partners --step 1   # ポータルサイト探索のみ
/collect-ses-partners --step 2   # 企業リスト収集のみ
/collect-ses-partners --step 3   # お問い合わせURL取得のみ
/collect-ses-partners --step 4   # シート統合のみ
```

### 実行時の進捗確認

- 各ステップ完了後、スプレッドシート上でリアルタイムに反映
- ポータル管理シートのステータス列で進捗確認
- 作業用シートのE列・H列で各企業の処理状態を確認

### 推奨実行環境

- Claude Codeが起動しているターミナルセッション
- インターネット接続（Google検索・各種サイトアクセス用）
- Google Spreadsheetへのブラウザアクセス権限（Playwright MCP経由）

## 9. 制約・方針

### コスト

- コストは考慮しない。精度・網羅性を優先する。

### クローリングマナー

- アクセス間隔: 各リクエスト後に1〜3秒のウェイトを設定
- robots.txt: 尊重する（アクセス前に確認）
- 過負荷防止: 同一ドメインへの連続アクセスを避ける

### データ操作

- Google Spreadsheetの操作はPlaywright MCPによるブラウザ直接操作のみ
- GAS（Google Apps Script）は使用しない
- Google Sheets API・OAuth認証は使用しない
- 重複チェックはGoogle Spreadsheet関数を最大限活用する

### Spreadsheet大量データ操作の安定性対策

Playwright MCPでのSpreadsheet操作はブラウザ経由のため、数千件規模のセル読み書きでは不安定になりうる。以下の対策を設計に組み込む。

- **バッチ処理**: セル操作は一度に50〜100行単位で実行し、各バッチ間で保存待ちを挟む
- **CSVエクスポート/インポート補助パス**: 大量データの読み取りにはPlaywrightでCSVダウンロード → Pythonで加工 → Playwrightでペースト、というフローも選択可能にする
- **進捗チェックポイント**: 各バッチ完了後にスプレッドシート上の状態を確認し、書き込み漏れがないかを検証する
- **リトライ**: 書き込み失敗時は3回までリトライ、それでも失敗した場合はエラーログを出力して続行する

### 拡張性

- シート1の列構成は柔軟に変更可能とする
- 検索クエリは追加・変更可能
- ポータル管理シートにより、クロール済みサイトを自動スキップして重複作業を防止
